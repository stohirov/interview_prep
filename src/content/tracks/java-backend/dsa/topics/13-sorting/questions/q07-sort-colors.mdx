**Problem.** Given an array of `n` objects colored red, white, or blue (encoded `0`, `1`, `2`), sort them in place so colors are adjacent in the order red, white, blue. Do it in one pass without a library sort.

## Brute force — counting sort, two passes, O(n)

Count each color, then overwrite. Correct and O(n), but it's **two passes** and the interviewer is fishing for the one-pass partition.

```java
void sortColorsCount(int[] nums) {
    int[] c = new int[3];
    for (int x : nums) c[x]++;
    int i = 0;
    for (int v = 0; v < 3; v++)
        while (c[v]-- > 0) nums[i++] = v;
}
```

## Optimal — Dutch National Flag, one pass, O(1) space

Maintain three regions with three pointers: everything before `low` is `0`, everything after `high` is `2`, and `[low, mid)` is `1`. The pointer `mid` scans; we shrink the unknown middle region to nothing.

```java
void sortColors(int[] nums) {
    int low = 0, mid = 0, high = nums.length - 1;
    while (mid <= high) {
        switch (nums[mid]) {
            case 0 -> swap(nums, low++, mid++);  // 0 -> front; both advance
            case 1 -> mid++;                     // 1 -> already in place
            case 2 -> swap(nums, mid, high--);   // 2 -> back; DON'T advance mid
        }
    }
}

private void swap(int[] a, int i, int j) { int t = a[i]; a[i] = a[j]; a[j] = t; }
```

**O(n)** time, single pass, **O(1)** space.

<Callout kind="warn" title="The one bug everyone hits">
On a `2`, swap with `high` and decrement `high` but **do not advance `mid`** — the value just pulled in from `high` is unexamined and could itself be a `0` or `2`. On a `0` you *do* advance `mid`, because the value coming from `low` has already been scanned (it can only be a `1`, since `low ≤ mid`).
</Callout>

<ProbeNote>This is Dijkstra's three-way partition — the same logic that powers three-way quicksort on inputs with many duplicate keys. Articulate the loop invariant (the four regions) and explain precisely why `mid` advances on `0`/`1` but not on `2`. That invariant reasoning is the senior signal.</ProbeNote>
